# 项目定位

本项目是一个“空间型协作系统（Space Collaboration System）”。

它不是传统的任务管理工具，也不是以即时对话为中心的聊天应用，而是一个围绕 **Space** 展开的长期协作系统。系统关注的是：用户如何在一个持续存在的协作容器中推进目标、表达当前状态、沉淀过程、管理分支演化，并与 Agent 建立可持续、可控的协作关系。

在本项目中，**Space** 不是单次会话或临时任务集合，而是用户围绕某个主题、目标或长期事项持续工作的核心载体。一个 Space 应当能够承载该事项的上下文、状态、节点结构、历史过程、阶段产物、外部同步信息，以及与 Agent 协作所产生的可追踪结果。

---

# 项目目标

本项目的目标是构建一个面向长期推进过程的协作系统，使用户能够在同一个 Space 中持续组织、推进和回顾复杂事项，而不是将工作割裂在聊天记录、任务列表、零散文档和外部工具之间。

项目希望逐步建立以下能力：

- 让用户能够围绕某个长期事项创建并维护稳定存在的 Space
- 让当前状态、关键节点、历史过程和阶段成果得到清晰表达
- 让主线、分支、回流和阶段演化成为系统内可理解、可组织的结构
- 让 Space 外的重要进展可以被识别、筛选并同步回 Space
- 让 Agent 作为 Space 内的协作者参与推进，而不是脱离上下文的通用聊天助手
- 让用户对自己的 Space、历史、记忆和 Agent 行为边界拥有明确、可控的管理能力

# 核心问题

本项目试图解决的核心问题包括：

## 1. 长期事项缺乏稳定协作容器

用户在推进复杂事项时，往往需要同时处理目标、上下文、任务、文档、讨论、决策和历史，但传统工具通常只能覆盖其中一部分，导致长期事项缺少统一承载体。

## 2. 当前状态难以被准确表达

很多系统擅长记录“做了什么”，却不擅长表达“现在处于什么状态、卡在哪里、下一步是什么、哪些内容仍待判断”，导致推进过程难以持续。

## 3. 历史过程缺少结构化组织

真实协作过程并不是线性的。它通常包含主线推进、分支探索、阶段回退、结论回流和多轮修正。如果历史只能按时间平铺，系统就难以支撑复杂事项的长期管理。

## 4. 空间内外信息难以汇聚

用户的重要进展往往发生在 Space 之外，例如外部讨论、外部文档、异步结论或外部执行结果。系统需要为这些信息回流提供合适机制，而不是让 Space 逐渐失真。

## 5. Agent 缺少明确协作边界

在很多系统中，Agent 只是平台级对话接口，缺少长期上下文、缺少边界、缺少可控性。本项目希望 Agent 成为 Space 内的协作者，其行为、记忆范围和作用边界都应清晰可控。

# 核心概念

## Space

**Space** 是系统中的核心协作对象。

它代表一个长期存在的协作容器，用于承载某个事项、目标、主题或持续推进中的工作域。Space 应能够容纳当前状态、节点结构、历史过程、阶段成果、外部同步信息和 Agent 协作关系。

Space 是用户长期推进工作的第一性对象，而不是任务、聊天或文档的附属集合。

## Node

**Node** 是 Space 内的结构化工作单元。

Node 可用于表达目标、任务、问题、决策、文档、里程碑、待办事项或其他需要被组织和推进的内容。Node 是 Space 内部结构和工作流的基本构件，应支持被组合、关联、演化和回溯。

## Timeline

**Timeline** 用于表达 Space 的过程性历史。

它不是简单的时间记录，而是对推进过程的组织方式。Timeline 关注的是：某个 Space 如何从一个状态演化到另一个状态，经历了哪些关键事件、阶段、分支、回流和结果沉淀。

## Snapshot

**Snapshot** 表示某个时间点或阶段下，Space 的相对稳定状态切片。

它用于帮助用户理解“当时系统的主要状态是什么”，可作为回顾、比较、分支判断、阶段归档和恢复上下文的重要依据。

## Sync Candidate

**Sync Candidate** 表示发生在 Space 外、但可能应被同步回 Space 的候选进展。

这类内容可能来自外部讨论、文档变化、执行结果、外部系统状态或其他协作上下文。系统不应默认无差别吸收所有外部信息，而应支持识别、筛选和确认哪些外部进展值得进入 Space。

## Agent

**Agent** 是 Space 内的协作者，而不是平台级通用聊天助手。

Agent 的作用应围绕具体 Space 的上下文展开，例如参与分析、推进、整理、总结、生成产物或协助同步信息。Agent 的记忆范围、行为边界、可见上下文和执行权限都应被清晰定义，并保持用户可控。

# 系统边界

本项目的总体方向是构建一个长期协作系统，而不是覆盖所有协作软件场景。

## 项目关注的内容

- 长期存在的 Space 及其内部结构
- 当前状态、阶段、节点、历史和产物的持续组织
- 主线、分支、回流和阶段演化的表达
- 外部进展向 Space 的同步与吸收
- Agent 在 Space 内的协作角色与边界控制
- 用户对 Space、历史、记忆和 Agent 的可控性

## 当前明确不作为核心目标的内容

- 传统意义上的通用即时聊天产品
- 仅以任务清单为核心的项目管理工具
- 平台级、脱离上下文的统一 AI 助手
- 当前阶段即实现完整权限体系、完整知识图谱、完整向量检索、完整外部同步网络
- 以短期功能堆叠替代长期结构设计

系统将优先围绕核心协作对象与长期推进过程建立能力，而不是先追求功能广度。

# 当前阶段

项目当前处于 **MVP 基础工程骨架阶段**。

此阶段的重点不是实现完整业务功能，而是先形成一个可持续迭代的稳定工程基线，为后续能力建设提供明确边界和可靠基础。

当前阶段重点包括：

- 建立 monorepo 工程结构
- 初始化 `web`、`core-api`、`ai-service` 三个应用
- 建立本地开发基础设施
- 为后续 Space、Node、Timeline、Agent 等能力预留清晰模块边界
- 形成规范驱动、可追踪、可持续演进的开发基线

近期目标是：

- 完成基础应用骨架与本地依赖联通
- 固化主要工程边界和目录结构
- 建立后续功能演进所需的规范、设计与任务分解机制
- 避免在基础阶段过早引入复杂业务实现和过度设计

当前阶段仍以结构建立、边界清晰和稳定基线形成为优先目标，暂不以功能完备度作为主要推进标准。

# 技术方向

项目当前采用的总体技术方向如下：

- 前端：React + TypeScript + Vite + Ant Design
- 主业务后端：Node.js + NestJS + Prisma
- AI 编排层：Hono
- 数据库：PostgreSQL
- 缓存与队列：Redis + BullMQ
- 本地开发：Docker Compose

这一技术方向服务于以下目标：

- 保持前端和后端边界清晰
- 让主业务服务与 AI 编排层分离演进
- 为长期数据存储、状态组织和历史能力提供稳定基础
- 为后续异步任务、事件处理和 Agent 协作预留基础设施条件
- 保障本地开发、联调和后续扩展的可持续性

当前架构采用“主业务服务 + AI 编排服务”分离方式推进，其中主业务服务承载核心领域对象与系统状态，AI 编排服务承载模型调用与智能协作能力。

当前文档只说明技术方向，不固化过细实现细节。具体实现约定应在相应的设计文档、规范文档和变更文档中进一步展开。

# 开发与演进方式

本项目采用 **规范驱动开发（spec-driven development）** 方式推进，并围绕 OpenSpec 进行受控演进。

这意味着：

- 重要变更应先被清晰描述，而不是直接进入无边界实现
- 项目演进以 proposal、design、specs、tasks 等产出物为基础
- 架构、边界、能力和任务需要保持可追踪、可解释、可归档
- 变更应尽量以受控方式推进，避免无记录扩散和结构污染
- 当前基础工程、后续能力建设和长期方向调整，都应逐步沉淀为规范资产

OpenSpec 在本项目中的作用，不只是记录变更，更是帮助项目维持长期结构一致性、边界清晰度和演进连续性的核心机制。
