## ADDED Requirements

### 需求:前端应用基础运行骨架
系统必须提供一个基于 React、TypeScript 和 Vite 的前端应用骨架，并完成 Ant Design、React Router、axios、TanStack Query 与 Zustand 的基础接入。

#### 场景:启动前端开发服务
- **当** 开发者启动 `web` 应用
- **那么** 应用必须能够以基础工程形态运行并加载前端依赖

### 需求:前端目录分层
系统必须采用 `app / pages / features / shared` 的前端目录分层，并保持各层职责清晰。

#### 场景:查看前端源码结构
- **当** 开发者浏览 `web/src` 目录
- **那么** 必须能够识别应用壳层、路由页面层、领域模块层与共享层的分层边界

### 需求:应用壳层与导航布局
系统必须提供稳定的应用布局层，用于承载主导航、页面容器和后续空间型协作页面入口。

#### 场景:访问前端根路由
- **当** 用户打开前端应用
- **那么** 页面必须呈现基础布局和明确的主导航区域

### 需求:页面层保持轻量
系统必须让 `pages/` 只负责路由级编排、读取路由参数和组装 feature 容器，不得在页面组件中直接堆放复杂业务逻辑或散落 HTTP 请求。

#### 场景:检查路由页面实现
- **当** 开发者查看任一路由页面组件
- **那么** 页面代码必须以布局和 feature 组装为主，而不是直接处理大量请求与状态细节

### 需求:占位路由页面
系统必须为核心页面提供路由占位，包括但不限于空间、节点、时间线、仪表盘等后续扩展入口。

#### 场景:切换到占位页面
- **当** 用户访问任一已定义的占位路由
- **那么** 系统必须展示与该路由对应的页面占位内容，而不是空白页或未处理错误

### 需求:Feature 内聚请求与服务端状态逻辑
系统必须让各领域 feature 在自身目录内维护 `api.ts`、`hooks.ts`、`types.ts` 以及需要的 `components/`、`containers/`、`utils.ts`，并通过 feature hooks 管理服务端状态。

#### 场景:检查领域模块结构
- **当** 开发者查看 `space`、`node`、`timeline`、`dashboard` 或 `auth` 等 feature 目录
- **那么** 必须能够识别一致的内部结构，并看到请求函数与 Query hooks 优先收敛在 feature 内部

### 需求:共享层只承载跨领域复用能力
系统必须让 `shared/` 仅承载真正跨页面、跨 feature 的 UI 组件、工具函数、通用 hooks、常量和 HTTP 基础封装，不得放入具体业务逻辑。

#### 场景:检查共享层内容
- **当** 开发者查看 `shared/` 目录
- **那么** 共享层内容必须表现为跨领域复用能力，而不是特定业务场景实现

### 需求:样式基线不引入 Tailwind
系统必须以 Ant Design 和 Less 作为当前阶段的样式基线，不得把 Tailwind 作为当前 MVP 基线依赖。

#### 场景:检查前端样式依赖
- **当** 开发者查看 `web` 的依赖与样式入口
- **那么** 必须能够确认样式体系以 Ant Design 和 Less 为主，且不存在 Tailwind 基线依赖
