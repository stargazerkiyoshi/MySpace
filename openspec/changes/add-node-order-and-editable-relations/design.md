## 上下文

当前节点图的数据来自两个不同来源：节点本身提供标题、状态和内容，Timeline 事件提供自动推导的边关系。这个 MVP 方案能快速出图，但它有两个明显缺口：一是节点没有稳定顺序，图上的左右位置会受列表排序和更新时间影响；二是边关系完全不可编辑，用户无法把图整理成自己想表达的结构。

这次变更会同时涉及 Prisma 数据模型、`core-api` 的图数据接口和 `web` 的 React Flow 交互，因此需要先明确一个最小但稳定的设计。

## 目标 / 非目标

**目标：**
- 给每个 Node 增加稳定顺序，并在同一个 Space 内按顺序返回和布局。
- 给 Graph View 增加可持久化的节点关系模型，支持删除已有连线和手动新建连线。
- 保持当前 Space 首页主画布不变，只增强画布数据和最小编辑能力。
- 避免再把 Timeline 当作图关系的唯一真相源。

**非目标：**
- 本次不做完整的拖拽排序 UI。
- 本次不做多种关系类型编辑，也不引入复杂关系属性。
- 本次不重做 Timeline 结构模型；Timeline 仍然负责历史记录，不承担图编辑真相。

## 决策

### 1. 给 Node 增加 `orderIndex`，创建时按当前最大值递增
- 决策：在 `Node` 表上新增 `orderIndex Int`，同一 Space 内唯一；创建节点时自动追加到末尾。
- 理由：用户先需要“明确前后顺序”，而不是复杂排序器。用显式顺序字段最直接，也最稳定。
- 替代方案：
  - 继续用 `createdAt` 或 `updatedAt` 推导顺序。
  - 放弃原因：会继续出现“更新后跳位”或“新节点在左边”的问题。

### 2. 新增独立 `NodeRelation` 模型作为图连线真相源
- 决策：新增 `NodeRelation` 表，仅记录 `spaceId / sourceNodeId / targetNodeId / relationType / createdAt`，首版 `relationType` 固定为 `sequence`。
- 理由：用户要求删除和重连连线，这已经是显式结构编辑，不能再依赖 Timeline 推导。
- 替代方案：
  - 把可编辑边继续写回 Timeline。
  - 放弃原因：Timeline 是历史事件流，不适合作为用户当前结构关系的编辑存储。

### 3. 图接口优先返回 `NodeRelation`，旧 Timeline 推导边不再参与图渲染
- 决策：`/spaces/:spaceId/node-graph` 只返回显式节点关系；如果没有关系则返回空边集。
- 理由：一旦允许用户编辑边，图上的边必须唯一、稳定、可解释。
- 替代方案：
  - 混合返回显式边和 Timeline 推导边。
  - 放弃原因：会再次出现重复边、优先级不清和删除后仍然“冒出来”的问题。

### 4. Web 端用 React Flow 原生交互承接删除和重连
- 决策：保留现有 React Flow 画布，新增 `onConnect` 创建边、节点/边删除、边去重和刷新逻辑。
- 理由：当前依赖已经具备最小交互能力，不需要新引入前端图编辑库。
- 替代方案：
  - 自建覆盖层按钮或单独编辑模式。
  - 放弃原因：实现更重，而且会让 MVP 首页交互变复杂。

## 风险 / 权衡

- [风险] 旧 Space 里的 Timeline 推导边在切换到 `NodeRelation` 后会消失
  缓解：本次接受这一收敛，图关系真相改为显式编辑；新关系由用户建立。
- [风险] 没有排序 UI 时，`orderIndex` 只能保证创建顺序
  缓解：先满足“知道谁在前”，后续再补手动调整顺序能力。
- [风险] 删除边可能让局部图失去可读性
  缓解：允许用户立即重新连线，并在接口层禁止重复边和自环边。

## Migration Plan

- Prisma 迁移新增 `Node.orderIndex` 和 `NodeRelation`。
- 为已有 Node 回填顺序值，按 `createdAt ASC` 生成。
- 图接口切换到显式关系模型。
- 前端画布接入创建/删除关系请求并刷新图。

## Open Questions

- 节点顺序后续是否需要独立 UI 进行上移/下移，当前暂不纳入本次实现。
- 首版是否允许多种关系类型，当前先固定为 `sequence`。
