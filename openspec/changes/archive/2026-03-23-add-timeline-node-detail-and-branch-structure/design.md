## 上下文

当前系统已经完成时间线基础展示：Node 的关键变化会写入 TimelineEvent，并在 Space 页面中显示最近事件列表。但现有 Timeline 仍然以“事件列表”视角为主，缺少节点详情层与基础结构层，用户无法理解某个节点是主线推进还是分支产物，也无法从时间线本身看出它与前后节点、当前状态之间的关系。

本次变更的重点不是实现完整历史树，而是在现有 TimelineEvent 基础上建立最小结构语义和最小详情交互。现有约束包括：

- 历史系统必须继续服务于 Space 内长期推进，而不是演化成装饰性时间轴。
- 时间线结构信息必须进入稳定数据模型，而不是只停留在前端硬编码。
- 当前前端仍然遵守 `app / pages / features / shared` 分层，时间线相关交互收敛在 `features/timeline`。
- 当前后端仍然遵守 `controller -> application/use case -> infrastructure` 链路，不在 controller 中直接拼装 Prisma 查询。

## 目标 / 非目标

**目标：**
- 为 TimelineEvent 增加主干 / 分支基础结构字段，使其能表达节点归属和前后关系。
- 为时间线节点提供最小详情能力，让用户点开后能理解节点含义、关系和影响。
- 在时间线列表展示中让主干节点、分支节点和关键节点具有可感知差异。
- 为未来 merge / 回归主线 / 当前状态联动预留稳定结构，但不提前实现完整逻辑。

**非目标：**
- 不实现完整历史树、脑图式交互或复杂时间线图谱。
- 不实现完整分支合并逻辑、自动回归主线或多层结构编辑器。
- 不实现 Agent 自动写入历史、外部系统同步时间线或首页双向跳转闭环。
- 不把 Timeline 一次性扩展成完整状态演算引擎。

## 决策

### 1. 时间线节点继续基于 TimelineEvent 扩展，而不是引入第二套历史节点对象
- 决策：在现有 TimelineEvent 模型上补充结构字段和详情字段，不新建并行的“HistoryNode”对象。
- 理由：当前 TimelineEvent 已经承担了 Space 内历史记录入口，继续扩展它可以保持演进连续性，避免短期内引入两套历史对象。
- 替代方案：
  - 新建单独的 TimelineNode/HistoryNode 模型。
  - 放弃原因：会造成事件记录与节点详情语义分裂，增加迁移成本。

### 2. 主干 / 分支先用显式字段表达，而不做树状交互
- 决策：第一版使用 `isMainline`、`parentNodeId`、`branchFromNodeId`、`mergeToNodeId`、`nodeType` 明确结构语义。
- 理由：这些字段足以支持未来分支再分支、回归主线和主干筛选，也符合“先稳定模型，再扩展交互”的原则。
- 替代方案：
  - 只在前端根据顺序和颜色假定主线/分支。
  - 放弃原因：结构信息无法成为稳定真相源，后续演进会受阻。

### 3. 节点详情优先采用 Space 页面内联详情区或侧边区域
- 决策：时间线节点详情由现有 Space 页面结构承载，采用最小详情区、侧边面板或等价交互，不引入独立时间线详情路由。
- 理由：当前时间线能力仍依附于 Space 上下文，用户应在同一工作区内理解节点与当前状态的关系。
- 替代方案：
  - 跳转到独立 Timeline Detail 页面。
  - 放弃原因：会打断当前 Space 内协作流，也会提前引入更重的路由复杂度。

### 4. 当前状态关系先通过解释性字段和展示表达承载
- 决策：节点详情中增加“对当前状态的影响”“是否通向当前主线”“前置 / 后继节点”等最小关系表达。
- 理由：本次目标是“理解历史”，不是“实现完整状态机”；解释性关系足以支撑最小理解闭环。
- 替代方案：
  - 直接做首页状态与时间线节点的双向联动。
  - 放弃原因：范围过大，会把本次变更从结构升级扩大成跨页面状态系统。

## 风险 / 权衡

- `[风险] TimelineEvent 承担更多结构职责后，字段设计可能需要后续迭代`
  → `缓解：本次明确先收敛最稳定的结构字段，避免一次性引入完整树状模型。`
- `[风险] 时间线详情信息过多会让 Space 页面变重`
  → `缓解：详情先聚焦最小信息可读性，不扩展为完整编辑器。`
- `[风险] 主线 / 分支的视觉表达如果过强，可能被误解为完整树状系统`
  → `缓解：本次只做到可感知区分，不承诺完整树状可视化。`
- `[风险] mergeToNodeId 预留但暂不使用，容易产生“字段已存在但功能未完成”的理解偏差`
  → `缓解：在设计和规范中明确其为未来预留字段，不作为本次交互闭环的一部分。`

## Migration Plan

- 在 TimelineEvent 模型中新增节点类型、主线/分支关系和详情展示所需字段，并生成迁移。
- 调整 `core-api` 的 timeline 查询与 DTO，使其返回结构关系和详情展示所需的最小字段。
- 在 `web` 的 `features/timeline` 中补充节点详情状态、主线/分支样式映射和详情视图。
- 在 Space 页面中接入时间线详情交互，并验证用户可以区分主线 / 分支并理解节点关系。

## Open Questions

- 第一版“对当前状态的影响”更适合直接存储摘要文本，还是从 payload 派生，当前倾向于保留显式展示字段。
- 当前节点 / 关键节点的强调规则是否完全由后端返回，还是前端根据结构字段推导，当前倾向于由前端基于结构字段先做最小推导。
- 前置节点 / 后继节点是否只展示单跳关系，当前倾向于第一版只展示直接相邻关系。
