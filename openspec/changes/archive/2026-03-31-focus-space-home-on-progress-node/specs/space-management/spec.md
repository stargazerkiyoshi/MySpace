## 新增需求

### 需求:Space 页面必须默认聚焦当前进展节点
系统必须在用户进入某个 Space 页面时优先展示当前进展节点或最接近当前推进主线的节点，而不是默认同时展开多个并列信息区。

#### 场景:进入某个 Space 时看到当前进展节点
- **当** 用户进入某个具体 Space 的页面
- **那么** 页面必须先展示一个明确的当前进展节点主视图
- **并且** 该主视图必须优先承载用户继续推进当前 Space 所需的核心上下文

## 修改需求

### 需求:Space 详情页必须承载 Timeline 区域
系统必须在 Space 详情页或 Space 首页中承载该 Space 的 Timeline 区域，以便用户看到最近发生的关键变化；当首页状态与历史形成联动后，该页面还必须承载当前状态与历史来源之间的最小联动入口；当 Space 开始支持结构可视化后，该页面还必须与 Node Graph View 一起构成同一 Space 工作区内的多视角表达；但这些次级信息默认不得压过当前进展节点主视图。

#### 场景:进入某个 Space 后查看结构化 Timeline
- **当** 用户进入某个具体 Space 的详情页
- **那么** 页面必须展示该 Space 的 Timeline 列表区域
- **并且** 该区域必须能够让用户感知节点是否属于主干或分支

#### 场景:从当前状态进入相关历史节点
- **当** 用户在 Space 首页或当前状态区域查看某个历史来源摘要
- **那么** 页面必须提供进入相关历史节点的真实入口
- **并且** 用户进入后必须能继续在同一 Space 上下文中理解该节点与当前状态的关系

#### 场景:次级信息默认收起
- **当** 用户首次进入某个 Space 页面
- **那么** Space 详情、最近变化和其他次级信息必须默认处于收起、隐藏或次级入口状态
- **并且** 页面不得让这些次级信息在默认状态下与当前进展节点主视图并列争夺首屏注意力
