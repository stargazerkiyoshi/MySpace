## 上下文

当前系统已经具备以下基础：

- Space 首页 / 详情页已经承载 Node 区域与 Timeline 区域
- TimelineEvent 已具备主干 / 分支基础结构字段与节点详情能力
- 用户已经能在时间线中查看某个历史节点的标题、类型、摘要、前置 / 后继关系和对当前状态的基础说明

但系统仍缺少“当前状态”和“历史来源”之间的稳定挂接语义。首页展示的是当前状态，时间线展示的是过去发生过什么，这两套信息仍然是并列而非联动的。结果是用户虽然能分别看到“现在”和“过去”，却难以理解当前主线来自哪段历史、哪个关键节点仍在影响现在，以及哪些分支已经偏离当前状态。

本次变更的重点不是完整路径演算，而是在现有 Space / Timeline 基础上建立最小的当前状态锚点语义和首页到历史的联动入口。

## 目标 / 非目标

**目标：**
- 建立当前状态与时间线历史节点之间的最小稳定挂接语义。
- 让首页状态区域能够展示“当前为什么会是这样”的最小历史来源信息。
- 让用户可以从首页状态区域进入相关历史节点。
- 让时间线节点详情能够表达该节点是否仍影响当前状态，以及它与当前主线的关系。
- 为后续主线路径追踪、完整双向回溯和 Agent 自动总结预留稳定结构。

**非目标：**
- 不实现完整的历史路径回放。
- 不实现完整的主线演化链自动计算。
- 不实现复杂分支合并影响分析。
- 不实现 Agent 自动生成历史节点或自动入线。
- 不实现首页与历史页的全量双向导航体系。

## 决策

### 1. 当前状态与历史的关系先通过独立读模型语义承载，而不是一次性引入完整持久化状态系统
- 决策：新增 `current-state-linking` 能力，在接口与 ViewModel 层明确以下语义：`currentMainlineNodeId`、`currentStateSourceNodeId`、`latestMainlineEventId`、`latestBranchEventId`、`isAffectingCurrentState`、`impactType`。
- 理由：当前系统已经有 TimelineEvent 作为历史真相源，但还没有完整的首页状态聚合模型。先用稳定读模型语义把“当前状态从哪里来”表达清楚，成本更低，也更适合 MVP 阶段。
- 替代方案：
  - 直接新增完整的 CurrentState 持久化模型。
  - 放弃原因：当前阶段会把范围从“状态与历史联动”扩大成新的核心领域对象设计，过重。

### 2. 首页状态来源优先围绕“关键锚点”组织，而不是一次性展示整条历史路径
- 决策：首页状态区域只要求展示少量关键历史来源，例如当前状态来源节点、最近关键推进节点、最近关键分支节点和当前主线锚点。
- 理由：用户当前最需要的是“现在从哪里来”的解释，而不是完整路径图。先收敛到几个高价值锚点，可以建立稳定闭环并减少页面负担。
- 替代方案：
  - 直接在首页内绘制完整路径或历史树。
  - 放弃原因：会过早引入重型可视化和路径计算，超出本次范围。

### 3. 时间线节点是否影响当前状态必须由稳定映射层产出，而不是纯前端硬编码判断
- 决策：在后端 timeline / current-state-linking 读模型层统一映射 `isAffectingCurrentState`、`impactType`、`isCurrentStateSource`、`isCurrentMainlineAnchor` 等关系字段，再由前端消费。
- 理由：如果这层关系只靠前端根据零散字段推断，后续接入 Snapshot、路径追踪或 Agent 自动总结时会出现重复逻辑和行为漂移。
- 替代方案：
  - 由前端直接根据 `isMainline`、`parentNodeId` 等字段推导。
  - 放弃原因：推导规则会散落在 UI 层，不利于后续扩展。

### 4. 首页与时间线详情继续承载在现有 Space 页面体系中，不新增独立“状态追溯页”
- 决策：本次首页状态来源卡片和时间线详情联动继续放在现有 Space / Dashboard 承载体系中，主要通过卡片摘要、链接入口和局部面板实现。
- 理由：当前用户工作流仍围绕 Space 页面展开，新建独立状态追溯页只会打断当前上下文。
- 替代方案：
  - 新增独立的 State Trace 页面。
  - 放弃原因：路由和信息架构成本过高，且不利于最小闭环。

## 风险 / 权衡

- `[风险] 当前状态锚点语义可能在后续 Snapshot 或路径追踪接入后需要调整`
  - `缓解：本次优先收敛为读模型字段和映射层语义，避免过早固化底层持久化结构。`

- `[风险] 首页状态区域加入历史来源后可能变重`
  - `缓解：只承载少量关键来源卡片与明确入口，不扩展为完整历史浏览器。`

- `[风险] “是否仍影响当前状态”在 MVP 阶段无法做到完整精确推理`
  - `缓解：本次允许先基于主干 / 分支锚点和当前状态来源映射输出最小可解释结果，但要求关系字段由稳定映射层统一产出。`

- `[风险] 首页与时间线都展示关系信息，容易出现重复`
  - `缓解：首页负责“当前从哪里来”，时间线详情负责“该节点如何影响现在”，两者职责分开。`

## Migration Plan

- 在 Timeline 现有结构基础上补充当前状态关联读模型与映射字段。
- 为首页状态区域补充最小历史来源读取能力与跳转入口。
- 为时间线节点详情补充“是否影响当前状态 / 影响类型 / 当前主线关系”字段和展示。
- 更新相关规范和 README，明确当前状态锚点语义与最小联动闭环。

## Open Questions

- 当前首页状态区域是继续沿用现有 dashboard 占位承载，还是在 Space 首页内优先收敛为“当前状态摘要卡片”，需要结合当前页面结构决定具体落点。
- `activeBranchId` 这类语义是否在本阶段必须显式返回，还是只返回最近关键分支节点即可，当前倾向于先返回关键分支锚点而不是完整 branch 对象。
- “已失效 / 被替代”这类影响类型在 MVP 阶段是由规则映射层输出，还是允许部分场景先不覆盖，当前倾向于先覆盖最常见的推进、偏移、中断、已失效四类。
