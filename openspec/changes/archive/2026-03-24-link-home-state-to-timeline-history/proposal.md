## 为什么

当前系统已经能分别展示首页状态、时间线列表和历史节点详情，但“现在”与“过去”之间仍然存在关键断层。用户能看到当前状态，也能看到历史发生了什么，却仍然难以理解当前主线来自哪段历史、哪些节点仍在影响现在，以及哪些分支已经偏离当前状态。

现在推进这项变更，是为了让首页状态页和时间线历史首次形成最小可追溯闭环，使系统从“状态与历史并存”升级为“状态与历史互相解释”。

## 变更内容

- 新增“当前状态与历史挂接关系”的最小能力，明确当前主线锚点、当前状态来源节点和节点是否仍影响当前状态等语义。
- 在首页状态区域增加历史来源表达，使首页不仅展示当前状态，还能说明当前状态与哪些历史节点相关。
- 提供从首页跳转到关键历史节点的最小真实入口，例如当前状态来源节点、最近关键推进节点或最近关键分支节点。
- 在时间线节点详情中补充“是否仍影响当前状态”“是否进入当前主线”“影响类型”等关系表达。
- 为后续主线路径追踪、完整双向联动和 Agent 自动总结接入预留稳定结构，但本次不实现完整路径回放、复杂分支分析或自动推理。

## 功能 (Capabilities)

### 新增功能
- `current-state-linking`: 定义首页当前状态与时间线历史节点之间的挂接关系、当前状态锚点语义以及最小双向联动入口。

### 修改功能
- `timeline-management`: 扩展时间线节点详情中的“影响当前状态”表达，并支持与当前主线 / 当前状态来源的关系展示。
- `space-management`: 要求 Space 首页 / 当前状态区域承载历史来源信息和跳转入口，使“当前状态”不再是孤立面板。
- `web-app-shell`: 调整首页状态页与时间线历史节点的交互承载方式，支持从当前状态进入相关历史节点。

## 影响

- `apps/core-api` 中与首页状态摘要、时间线查询和关系映射相关的读模型与接口。
- `apps/web` 中首页状态页、Space 页面、时间线详情交互和当前状态摘要展示。
- `openspec/specs/` 中 current-state-linking、timeline-management、space-management、web-app-shell 的规范。
