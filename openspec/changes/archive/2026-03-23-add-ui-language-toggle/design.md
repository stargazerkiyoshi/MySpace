## 上下文

当前 `web` 已经采用 `app / pages / features / shared` 分层，并完成基础壳层与 Space 最小业务闭环。现有文案主要直接写在组件中，尚未建立统一的语言状态与文案组织方式。如果继续在单语文案基础上扩展页面，后续补做国际化会快速演化为大面积散改。

当前约束包括：

- 现阶段只需要支持 `zh-CN` 与 `en` 两种 UI 语言。
- 不希望当前阶段引入过重的国际化框架或复杂编译链。
- 前端必须保持既有分层和 feature 内聚，不把文案逻辑散落回页面层。
- 当前只处理前端 UI 文案，不处理数据库内容、用户输入和业务数据翻译。

## 目标 / 非目标

**目标：**

- 为前端建立轻量且可持续扩展的中英文切换基础。
- 提供全局语言状态、统一语言切换入口和本地持久化能力。
- 覆盖应用壳层、导航、Space 页面和基础提示文案。
- 为后续页面继续接入双语提供统一写法。

**非目标：**

- 不引入完整企业级 i18n 平台或远程翻译管理。
- 不翻译数据库内容、用户输入和后端返回的原始业务字段。
- 不在当前阶段覆盖尚未实现的所有未来页面。

## 决策

### 1. 当前阶段采用轻量本地化方案

- 决策：不引入额外第三方 i18n 框架，先使用轻量文案字典 + 全局语言状态完成双语能力。
- 理由：当前页面规模有限，需求集中在 UI 文案切换，轻量方案更符合 MVP 阶段复杂度。
- 替代方案：
  - 直接引入 `i18next` / `react-intl`
  - 放弃原因：当前收益不足以覆盖集成与维护成本

### 2. 使用本地存储持久化语言选择

- 决策：当前语言通过前端全局状态维护，并同步到 `localStorage`。
- 理由：这可以满足刷新保留语言选择的需求，同时不依赖后端用户配置。
- 替代方案：
  - 只在内存中保存
  - 放弃原因：刷新后会丢失语言选择

### 3. 文案按“共享 + feature”分层组织

- 决策：应用壳层和跨页面共用文案放在共享层；领域相关文案放在对应 feature 内部。
- 理由：这符合现有前端架构，避免把所有文案堆成一个难维护的大文件。
- 替代方案：
  - 所有文案统一集中到单个全局文件
  - 放弃原因：随着 feature 增加会失去领域边界

### 4. 壳层提供统一切换入口

- 决策：语言切换入口放在应用壳层，所有已覆盖页面共享同一个入口。
- 理由：语言是应用级状态，不应为每个页面单独提供入口。
- 替代方案：
  - 在每个页面单独放切换按钮
  - 放弃原因：会导致体验和实现不一致

## 风险 / 权衡

- `[风险] 轻量方案后续可能演进成文案字典分散` → `缓解：从一开始明确共享与 feature 文案边界`
- `[风险] 页面内直接写死的新文案会绕过双语体系` → `缓解：在任务中显式要求把当前覆盖范围的文案全部改为字典读取`
- `[风险] 当前不翻译业务数据可能让页面出现双语 UI + 原始内容混排` → `缓解：在规范里明确这是当前阶段的边界，避免误解为缺陷`

## Migration Plan

- 增加语言状态与本地持久化基础。
- 在应用壳层增加语言切换入口。
- 将现有壳层、导航、Space 页面和公共提示迁移为双语文案。
- 完成后续页面时沿用同一套文案组织方式继续扩展。

## Open Questions

- 默认语言是否固定为 `zh-CN`，还是在无本地存储时按浏览器语言推断；当前倾向先固定中文，后续再扩展自动判断。
