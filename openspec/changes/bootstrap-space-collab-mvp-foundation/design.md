## 上下文

本次变更目标是在现有仓库中建立一个可持续扩展的空间型协作系统 MVP 工程基线。该基线需要同时覆盖三个独立运行单元:

- `web`: 面向用户的 React 前端应用
- `core-api`: 承载主业务领域能力的 NestJS 服务
- `ai-service`: 承载 AI 编排与对话入口的 Hono 服务

当前仓库尚未形成统一的 monorepo 结构、应用边界和本地依赖编排方式。若在此阶段直接堆叠业务能力，后续会在目录组织、共享配置、接口边界和本地开发体验上反复返工。

主要约束如下:

- 必须采用 `pnpm workspace` 管理 monorepo
- 必须为 PostgreSQL 与 Redis 提供本地开发支撑
- 必须保持 `core-api` 与 `ai-service` 的职责隔离，避免将 AI 编排逻辑直接耦合进主业务服务
- 当前范围只要求工程骨架与占位能力，不要求完整业务实现

## 目标 / 非目标

**目标：**

- 建立统一的 monorepo 目录结构和共享工程配置
- 让 `web`、`core-api`、`ai-service` 都能以最小骨架启动
- 在 `web` 中建立基础布局、主导航与路由占位页
- 在 `core-api` 中建立模块化 NestJS 骨架，并完成 Prisma/PostgreSQL/Redis 基础接入
- 在 `ai-service` 中建立基础 HTTP 服务与健康检查、聊天占位接口
- 提供可复用的本地开发基础设施与启动说明

**非目标：**

- 不实现完整认证、权限、空间协作、节点编辑、时间线、仪表盘业务逻辑
- 不实现复杂事件编排、任务队列消费、向量检索或外部系统同步
- 不在本次设计中定义完整领域模型、最终 API 契约或生产部署方案

## 决策

### 决策 1: 使用 `pnpm workspace` 建立多应用 monorepo

选择理由:

- `pnpm workspace` 适合统一管理多个 TypeScript 服务与共享配置，安装速度和磁盘效率也适合作为长期基线。
- 相比多仓库方案，monorepo 更适合在 MVP 阶段同步推进前端、主业务服务与 AI 服务。

备选方案:

- 多仓库拆分: 边界更强，但会增加联调、版本同步与脚手架维护成本。
- 引入更重的 monorepo 构建系统: 当前阶段收益不足，先保留简单 workspace 基线。

### 决策 2: 以应用边界而不是技术细节组织仓库

建议采用类似 `apps/web`、`apps/core-api`、`apps/ai-service` 的目录组织，并在根目录放置共享配置与基础设施文件。

选择理由:

- 应用边界清晰，便于明确责任归属和后续拆分共享包。
- 能避免在 MVP 初期过早引入大量共享库抽象，先把运行单元稳定下来。

备选方案:

- 按技术层拆分目录: 容易模糊运行单元边界。
- 立即创建大量 `packages/*`: 早期抽象成本较高，先只保留共享配置和必要公共类型扩展空间。

### 决策 3: 主业务能力集中在 `core-api`，AI 编排集中在 `ai-service`

选择理由:

- `core-api` 负责领域资源、认证、空间与节点等主业务模块，利于保持业务事务边界与数据一致性。
- `ai-service` 负责 AI 入口和编排扩展，利于后续独立演进模型接入、Prompt 编排和异步处理能力。
- 该边界可以避免“为了快”把 AI 能力直接写进主服务，导致职责混乱。

备选方案:

- 将 AI 接口直接并入 NestJS: 短期更快，但会污染主业务服务边界。
- 先不拆 AI 服务: 会推迟边界建立，后续迁移成本更高。

### 决策 4: `core-api` 使用 NestJS + Prisma，数据库为 PostgreSQL

选择理由:

- NestJS 适合模块化主业务服务骨架，便于组织 `auth`、`space`、`node`、`timeline`、`dashboard` 模块。
- Prisma 适合作为 MVP 阶段的类型安全 ORM 与 schema 管理工具。
- PostgreSQL 能够覆盖事务型主业务数据的长期需求。

备选方案:

- 轻量 Node 框架: 启动快，但对模块化大型服务的约束较弱。
- 直接手写 SQL/查询层: 初期灵活，但缺少统一 schema 演进和类型约束。

### 决策 5: Redis 作为基础设施能力先完成接入，不预先绑定复杂队列实现

选择理由:

- 当前范围只要求缓存/队列基础设施基线，先完成连接配置和服务编排即可。
- 避免在业务需求未明确前绑定特定消息流和消费模型。

备选方案:

- 同时引入完整队列框架: 会提前引入不必要复杂度。
- 暂不接入 Redis: 会让后续缓存和异步能力缺少稳定基础。

### 决策 6: 前端先建立应用 Shell 和路由占位，不提前沉淀复杂 UI 能力

选择理由:

- 先完成布局、导航、路由与数据访问基础，可为后续页面快速接入提供稳定入口。
- 使用 Ant Design、React Router、axios、TanStack Query、Zustand 作为稳定基线，覆盖 UI、路由、请求、服务端状态与本地状态的常见需求。

备选方案:

- 只初始化空白 Vite 项目: 无法表达后续页面结构与应用壳层。
- 在此阶段实现高保真业务页面: 超出当前范围，且易造成返工。

### 决策 7: 前端采用 `app / pages / features / shared` 分层，而不按纯技术类型堆叠目录

选择理由:

- `app` 只负责应用入口、全局 provider、Ant Design 配置、路由注册和应用级布局，不承载领域逻辑。
- `pages` 只承担路由级编排，避免页面组件直接堆放请求、状态和展示细节。
- `features` 以领域为边界内聚请求、hooks、局部组件、容器和类型，能让业务逻辑优先收敛在领域内部。
- `shared` 只保留跨领域复用能力，降低“通用层”被业务污染的风险。

备选方案:

- 仅使用 `components / hooks / utils` 这类纯技术目录: 早期上手快，但领域边界会迅速变得模糊。
- 在 MVP 阶段引入额外 `entities` 层: 当前领域模型尚未稳定，过早抽象会增加维护成本。

### 决策 8: 每个 feature 使用一致的内部结构，页面保持轻量

建议采用如下组织方式:

- `api.ts`: 仅定义当前领域的请求函数，基于共享 HTTP 封装发起请求
- `hooks.ts`: 封装 TanStack Query 的 `useQuery` / `useMutation` 和请求组合逻辑
- `types.ts`: 放当前领域的前端类型定义
- `components/`: 放展示组件和局部组合组件
- `containers/`: 在确有必要时承载数据获取、状态组合、事件分发
- `utils.ts`: 放当前领域内部纯函数

选择理由:

- 统一结构有助于在空间、节点、时间线、仪表盘、认证等领域之间保持可预测性。
- 页面层只负责组装 feature container，可以避免路由组件承担业务复杂度。

备选方案:

- 全部逻辑直接放在页面文件中: MVP 初期可行，但随着需求增长会迅速失控。
- 对所有 feature 机械拆出 container: 会制造不必要抽象，因此保持“需要时才使用”的约束更合理。

### 决策 9: 前端请求与状态管理固定采用 `axios + TanStack Query + Zustand`

选择理由:

- `axios` 负责 HTTP 实例、拦截器、基础错误处理和后续 token 注入扩展。
- `TanStack Query` 负责服务端状态缓存、重试、失效刷新和请求生命周期。
- `Zustand` 只承担必要的客户端状态，例如面板开关、当前选中节点和局部工作台状态。

约束:

- 页面组件不得直接散落 `axios` 调用。
- `features/*/api.ts` 只定义请求函数，不直接定义 React hook。
- `features/*/hooks.ts` 统一管理 queryKey、缓存失效和请求组合。
- 不得用 Zustand 维护后端列表数据真相，也不得把 Query 缓存与 Zustand 重复维护。

### 决策 10: 样式基线固定为 Ant Design + Less，不在当前阶段引入 Tailwind

选择理由:

- MVP 阶段优先保证结构清晰、一致性和可维护性，而不是同时维护两套样式范式。
- Less 足以承载 Ant Design 主题变量覆盖和少量需要集中维护的复杂样式。
- 避免在尚未稳定的阶段同时引入 Tailwind 原子类和 Less 规则，降低样式体系复杂度。

备选方案:

- Ant Design + Tailwind + Less 同时使用: 灵活但复杂度过高，不适合作为当前基线。
- 大量散落普通 CSS 文件: 会进一步削弱样式约束和工程一致性。

## 风险 / 权衡

- [风险] monorepo 初期未抽出共享包，后续可能出现少量重复配置
  → 缓解措施: 先统一根级 TypeScript、lint、环境变量与脚本约定，待重复稳定后再抽取共享包。

- [风险] Prisma schema 只做骨架，后续领域调整时可能发生迁移变化
  → 缓解措施: 本次只定义稳定的最小基础实体与扩展点，不提前固化复杂关系。

- [风险] AI 服务接口当前仅为占位，调用链尚不完整
  → 缓解措施: 在规范中明确其职责与边界，保证后续扩展在独立服务中进行。

- [风险] Docker Compose 只覆盖本地开发依赖，不等价于生产部署
  → 缓解措施: 在 README 中明确其用途仅限本地开发基线。

- [风险] pages、features 和 shared 之间职责边界随实现推进再次模糊
  → 缓解措施: 在规范和目录中固定分层约束，页面只编排，业务逻辑优先收敛到 feature。

## Migration Plan

1. 建立 monorepo 根目录与 workspace 配置。
2. 初始化 `web`、`core-api`、`ai-service` 三个应用与最小启动脚本。
3. 接入 PostgreSQL、Redis、Prisma 和基础环境变量模板。
4. 增加 Docker Compose 与 README，完成本地启动闭环。
5. 以后续 capability 为单位逐步填充实际业务逻辑。

回滚策略:

- 如任一应用初始化失败，可回滚到仅保留目录结构与配置文件的最小可用状态。
- 数据库部分仅包含初始 schema 骨架，回滚时删除本地开发容器与迁移文件即可，不涉及生产数据迁移。
