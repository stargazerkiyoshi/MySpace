## 上下文

当前 `SpaceDetailContainer` 进入页面后会先渲染 `SpaceDetailPanel`，再展示 `Nodes / Graph / Recent History` 三个并列 tab。虽然这些能力都有效，但首屏缺少明确的视觉主线，用户需要先理解多个板块，再自己判断“当前该看哪个节点、从哪里继续推进”。

这次变更的约束很明确：

- 保持现有 `app / pages / features / shared` 分层，不把页面编排逻辑散落到多个层级。
- 不新增独立路由，不把 Space 拆成更多子页面。
- 不新增新的底层读模型或关系模型，优先复用现有节点、当前状态和时间线能力。
- 次级信息不是删除，而是默认隐藏并提供明确展开入口。

## 目标 / 非目标

**目标：**
- 让用户进入某个 Space 时，第一眼就能看到当前进展节点或最接近当前推进主线的节点。
- 将 Space 详情、节点详情和最近变化从“默认同时展开”改为“显式触发后展开”。
- 让 Space 页面保留结构视图、历史视图和编辑能力，但不再与主视图争夺首屏注意力。

**非目标：**
- 不重做整个 Space 信息架构，不新增新的路由层级。
- 不在本次引入新的“主线判定算法”或新的后端聚合接口。
- 不移除 Graph View、Timeline 或 Space 编辑能力，只调整默认显隐和入口层级。

## 决策

### 1. 将 Space 页面首屏收敛为“当前进展节点主视图”
- 决策：进入 Space 后默认先看到单一主视图，优先展示当前进展节点，而不是默认展示空间元信息卡片加多个工作区 tab。
- 理由：用户进入页面时首先需要的是推进上下文，而不是全部能力目录。
- 替代方案：
  - 继续保留当前 “详情卡片 + tabs” 结构，仅做视觉样式优化。
  - 放弃原因：这只能改善美观，不能解决首屏注意力被分散的问题。

### 2. 次级信息改为抽屉、折叠或显式触发区，而不是默认并列
- 决策：Space 详情、节点详情、最近变化统一降级为按需展开的次级信息层；它们必须有明确入口，但默认不占据主视图。
- 理由：这能在不删除能力的前提下，保留页面清晰度。
- 替代方案：
  - 完全移除次级信息。
  - 放弃原因：会削弱空间管理、回顾历史和编辑内容的可达性。

### 3. 主视图优先复用现有节点与当前状态语义，不新增新接口
- 决策：首版进展节点优先从现有页面已可读取的数据中确定，例如当前状态关联节点、当前选中节点或列表中的首个推荐节点；若无法得到更强语义，也必须保持结果真实且可解释。
- 理由：本轮目标是主表达调整，不是扩展新的后端真相源。
- 替代方案：
  - 为“当前进展节点”单独新增后端字段或排名接口。
  - 放弃原因：会把一次页面收敛变成新的领域建模工作。

### 4. Graph View 作为二级视图保留，但不作为默认入口
- 决策：Graph View 保留在 Space 工作区中，但默认进入页面时不直接展示；用户需要通过明确入口切换或展开。
- 理由：Graph View 适合作为结构补充视角，而不是当前这条 UI 主线的首页主表达。
- 替代方案：
  - 直接以 Graph View 作为首屏。
  - 放弃原因：图视图更强调结构观察，不一定最适合表达“当前推进到哪里”。

## 风险 / 权衡

- `[风险] 现有数据对“当前进展节点”的判定语义仍偏弱`
  → `缓解：规范中明确首版允许基于现有真实数据做最小判定，但不得伪造新的推进语义。`
- `[风险] 将多块信息默认隐藏后，用户可能一时找不到详情或历史`
  → `缓解：所有次级信息都保留显式入口，并在主视图附近提供稳定触发位置。`
- `[风险] 页面收敛后，部分已有交互路径会改变`
  → `缓解：保持路由不变，并优先用抽屉、折叠和轻量切换承接旧能力。`

## Migration Plan

- 在 `space-management` 和 `web-app-shell` 中补充 Space 页面主视图优先与次级信息收起的规范。
- 在 `apps/web` 的 Space 页面与相关 feature 容器中，将默认主表达改为当前进展节点视图。
- 调整 Space 详情、节点详情、最近变化的默认显隐策略和入口位置。
- 验证中英文文案、默认进入体验以及展开次级信息后的可用性。

## Open Questions

- 当前进展节点的首版优先级应以“当前状态关联节点”为先，还是以“节点列表中的当前选中节点”为先，仍需结合实现阶段确认。
- 最近变化应采用抽屉、折叠区还是次级 tab 作为入口，当前倾向于轻量抽屉或折叠面板。
