# AGENTS.md

本文件定义本仓库中 AI 代理参与协作时应遵守的项目级规则。  
它面向支持 `AGENTS.md` 约定的工具，作为默认协作入口使用。

## 1. 项目定位

本项目是一个“空间型协作系统（Space Collaboration System）”。

它不是传统任务管理工具，也不是普通聊天应用，而是围绕 **Space** 展开的长期协作系统。每个 Space 是一个持续存在的协作容器，用于承载目标、当前状态、历史过程、任务节点、分支演化、文档产物，以及与 Agent 的长期协作关系。

如需理解项目背景、目标、核心概念与长期方向，优先阅读：

- [openspec/project.md](F:/projects/MySpace/openspec/project.md)

## 2. 当前阶段

当前处于 **MVP 基础工程骨架阶段**。

当前优先目标：

- 建立稳定的 monorepo 基线
- 维护 `web`、`core-api`、`ai-service` 三个应用的清晰边界
- 维护本地开发基础设施
- 为后续 Space / Node / Timeline / Agent 等核心能力预留稳定扩展位

当前不追求：

- 完整业务功能
- 复杂权限系统
- 完整 Agent 编排
- 向量检索
- 外部同步网络
- 完整历史树和图谱能力

## 3. 协作原则

AI 代理在本仓库中工作时，必须遵守以下原则：

1. 先明确方向，再进入实现  
   不要在需求、边界、层次不清晰时直接修改代码。

2. 先规范化，再工程化  
   涉及能力边界、结构变化、架构约束或长期演进的内容，应优先进入 OpenSpec 变更流程。

3. 变更必须受控  
   不要无记录地大范围重构、扩散修改或引入隐式约定。

4. 分层优先于功能堆叠  
   任何实现都应优先判断它属于哪一层，是否破坏现有边界。

5. 完成不等于写完  
   交付必须包括必要的验证、文档同步和结构一致性检查。

## 4. OpenSpec 工作方式

本项目采用 **规范驱动开发**，围绕 OpenSpec 做受控演进。

当工作涉及以下任一情况时，应优先考虑通过 OpenSpec 推进：

- 新能力引入
- 现有能力行为变化
- 架构调整
- 目录结构重组
- 模块边界变化
- 重要开发约定更新

OpenSpec 相关目录：

- `openspec/project.md`：项目总览与长期方向说明
- `openspec/changes/`：进行中的变更
- `openspec/specs/`：能力规范

默认流程：

1. 明确主线与边界
2. 收敛需求
3. 发起 OpenSpec 变更
4. 基于 proposal / design / specs / tasks 实现
5. 验证、归档、回到稳定基线

## 5. 目录与系统边界

当前系统主要由以下部分组成：

- `apps/web`：前端应用
- `apps/core-api`：主业务后端
- `apps/ai-service`：AI 编排层
- `openspec/`：规范与变更系统
- `docs/`：长期维护的项目文档

边界要求：

- `core-api` 承载主业务能力，不混入 AI prompt 编排逻辑
- `ai-service` 只承载 AI 编排入口与相关服务边界，不复制主业务模块
- 文档与架构约定优先沉淀到 `docs/` 或 `openspec/`

## 6. 前端协作规则

前端开发必须遵守独立架构文档：

- [docs/frontend-architecture.md](F:/projects/MySpace/docs/frontend-architecture.md)

关键约束摘要：

- 前端固定采用 `app / pages / features / shared` 分层
- 页面层保持轻量
- 领域逻辑优先收敛在 feature 内
- 请求层采用 `axios + TanStack Query`
- `Zustand` 只承载必要客户端状态
- 当前样式基线为 Ant Design + Less
- 当前阶段不引入 `entities` 层

## 7. 实现要求

进行代码实现时：

- 优先做最小、边界清晰、可验证的修改
- 不要为了“先跑起来”破坏分层
- 不要把短期便利写成长期结构
- 不要把未定型的业务逻辑提前固化成底层抽象

如遇到以下情况，应暂停并先澄清或更新规范：

- 需求含糊
- 实现会突破现有边界
- 需要引入新的跨层抽象
- 需要修改长期架构约定

## 8. 文档优先级

当多个文档同时存在时，优先参考顺序如下：

1. `openspec/project.md`
2. 当前活跃变更下的 `proposal.md / design.md / specs / tasks.md`
3. `docs/frontend-architecture.md`
4. `docs/开发规范.md`
5. 具体代码实现

如果实现与规范冲突，应优先指出冲突，而不是直接按现状继续扩散。

## 9. 输出风格

AI 代理输出应保持：

- 直接
- 清晰
- 结构化
- 可执行

避免：

- 空泛建议
- 无边界泛化
- 过度设计
- 在未说明取舍前擅自扩大范围

## 10. 一句话原则

先明确主线与边界，再通过 OpenSpec 发起受控变更，在保持分层清晰和系统稳定的前提下推进实现与验证。
