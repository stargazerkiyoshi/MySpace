# UI 模式与经验

本文件沉淀当前项目在前端界面收敛过程中的通用经验，用于指导后续页面与组件调整。

它不替代 [frontend-architecture.md](F:/projects/MySpace/docs/frontend-architecture.md) 的分层约束，而是补充“界面应该如何组织”和“什么样的表达更清爽”这类通用模式。

## 1. 先展示对象本体，不先解释页面是什么

很多页面天然已经带有明确上下文，例如：

- 空间列表页
- 空间详情页
- 节点页
- 最近变化页

在这类页面中，不要先堆一个重复标题去解释“这里是什么”。优先展示真正的对象本体和可操作内容。

推荐：

- 空间列表页直接展示“创建按钮 + 空间列表”
- 空间详情页直接展示“空间卡片 + 内容区”
- 节点页直接展示“节点列表 + 触发入口”

不推荐：

- 先放一层“空间 / 空间首页 / 节点 / 最近变化”的说明性页头
- 再进入真正内容

## 2. 控制容器层级，避免卡片套卡片

页面需要整体感，但不需要层层包裹。

推荐结构：

- 一个主对象卡片
- 一个主内容区
- 内容区内部尽量轻量

不推荐：

- 页面卡片
- 工作区卡片
- 分区卡片
- 列表卡片

当层级过深时，页面会变厚重、压迫，且很难读出重点。

## 3. 同类卡片必须统一尺寸规则

同类卡片一旦高低、宽窄、文案高度不一致，页面会立刻显乱。

推荐统一规则：

- 标题最多一行
- 说明最多两行
- 超出内容使用省略
- 悬停时展示完整内容
- 同类卡片使用相同最小高度
- 同类列表项使用相同纵向节奏

这条规则适用于：

- 空间列表卡片
- 首页历史锚点卡片
- 时间线列表项
- 节点列表项

## 4. Tag 只保留有判断价值的信息

Tag 的作用是突出“会影响理解和判断”的信息，而不是重复已有信息。

推荐保留：

- 主干 / 分支
- 节点状态
- 少数关键类型

不推荐保留：

- 页面上下文已经明确的信息
- 标题附近重复表达的信息
- 纯装饰性的类型标签

其他辅助信息更适合放到次级文字、说明行或元信息区域中。

## 5. 查看与编辑的交互方式按对象粒度决定

不同对象应采用不同交互方式，不要机械统一。

当前推荐：

- `Node`
  查看和编辑可以放在一起，用抽屉承载
- `Space`
  默认先看，编辑用模态框承载
- `Timeline`
  当前阶段优先只查看，不急着开放编辑

判断原则：

- 轻量对象、点开后常常顺手修改的内容，适合“查看 + 编辑”一体
- 全局对象、基础信息修改频率较低的内容，适合“查看”和“编辑”分层

## 6. 收缩态需要单独设计

收缩不是简单把宽度变窄。

对于侧栏、抽屉、卡片等会进入“紧凑模式”的区域，需要单独定义收缩态表达。

推荐：

- 侧栏收缩后仅保留品牌简称
- 不在收缩态继续渲染完整说明文案
- 避免在窄宽度中强行保留多行文字

不推荐：

- 让展开态标题直接在收缩态里换行
- 用“自动挤压”代替“收缩设计”

## 7. 页面清爽，更多来自减少重复信息

页面混乱通常不是因为功能太多，而是因为重复信息太多。

常见冗余包括：

- 重复页头
- 重复说明性文字
- 重复标签
- 重复元信息
- 重复容器

做 UI 收敛时，优先删除重复信息，再决定是否需要重新排版或换视觉样式。

## 8. 保留整体感，但不要堆厚重感

“清爽”不等于“完全打散”。

如果把所有外层容器都去掉，页面又会失去整体感。更稳的做法是：

- 保留一层真正有意义的整体结构
- 删除重复的内部包裹层

也就是说：

- 需要“整体”
- 不需要“厚重”

## 9. 调整顺序建议

当页面看起来乱时，建议按以下顺序处理：

1. 删除重复标题与说明
2. 删除无意义的 Tag
3. 降低容器层级
4. 统一同类卡片尺寸
5. 收紧留白与间距
6. 最后再调整颜色、阴影、边框等视觉细节

不要一开始就优先改视觉风格，否则容易在结构未清晰时反复返工。

## 10. 本项目当前适用范围

这些经验当前主要适用于：

- `apps/web` 中的列表页
- 空间详情与工作区页
- 节点、时间线、首页状态等内容区
- 弹层、抽屉、模态框的轻量信息组织

后续若引入更复杂的历史树、图谱、快照或 Agent 工作台，应在保持这些基本原则的前提下，按新场景补充新的 UI 模式。
