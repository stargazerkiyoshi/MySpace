# AI Service Architecture Baseline

本文件定义 AI 编排层子工程的通用架构基线，作为后续 `ai-service` 一类工程长期演进时的参考。

本文档关注的是 AI 编排服务的角色定位、边界、结构和演进约束，而不是某个具体项目当前阶段的业务规则。  
与具体 Space、Node、Timeline、Sync Candidate、Agent 行为策略相关的业务要求，应继续写入 OpenSpec 文档。

## 1. Scope

本基线适用于：

- 独立的 AI 编排服务
- 负责模型调用、上下文装配、工具调用和 Agent 执行流的服务
- 与主业务后端并行存在、但不替代其业务真相源角色的 AI 层

本基线不适用于：

- 主业务后端
- 仅提供单一模型代理的极简 SDK 包装层
- 不具备编排职责的纯前端 AI 调用模块

## 2. Role Of The AI Service

AI 编排层的定位是：

- 模型调用编排层
- Agent 执行协调层
- 上下文装配层
- 工具调用入口层
- 异步 AI 任务处理层

AI 编排层应负责：

- 构造模型输入上下文
- 组织 Prompt 与工具调用流程
- 管理 Agent 执行过程
- 处理 AI 结果的中间态、草稿态和候选态
- 与主业务后端协调读取数据与写回结果

AI 编排层不应负责：

- 成为业务真相源
- 直接管理主业务资源的最终状态
- 复制一套主业务后端模块
- 在没有明确业务契约时直接改写核心业务数据

## 3. Core Principles

### 3.1 AI Service Is Not A Second Backend

AI 编排层不是第二个业务后端。  
它不应复制 Space、Node、Timeline 等主业务模块的核心职责。

### 3.2 AI Output Is Candidate, Not Truth

模型输出、摘要、建议、计划、分析和同步结果，默认都应被视为候选结果，而不是业务事实。  
只有经过受控业务流程确认后，才应进入正式状态。

### 3.3 Explicit Context Is Better Than Hidden Memory

Agent 的上下文、记忆来源、工具权限和可写边界应显式定义。  
不要依赖不可追踪的隐式上下文拼装。

### 3.4 Tool Access Must Be Controlled

Agent 可以调用工具，但工具边界必须明确。  
工具调用能力不能成为绕过主业务后端边界的后门。

### 3.5 Orchestration Must Be Observable

AI 编排流程应尽可能可观察、可中断、可追踪。  
至少要能回答：

- 当前在执行什么
- 用了哪些上下文
- 调了哪些工具
- 输出了什么候选结果
- 最终是否写回了主业务系统

## 4. Recommended Structure

推荐采用如下结构：

```text
src/
  routes/
  orchestration/
  agents/
  providers/
  tools/
  memory/
  queue/
  contracts/
```

### `routes/`

职责：

- HTTP 入口
- 健康检查
- Chat / Agent / Task 等 API 入口

约束：

- routes 只负责请求接收与响应输出
- 不在 route handler 中直接堆编排逻辑

### `orchestration/`

职责：

- 编排流程
- 执行器
- 状态机
- 任务执行链路
- 上下文装配调度

约束：

- orchestration 负责“如何组织一次 AI 执行”
- 不在这里承载主业务真相

### `agents/`

职责：

- Agent 定义
- Agent 角色能力
- Agent 策略配置
- Agent 权限范围

约束：

- Agent 定义应显式说明其目标、可见上下文、可调用工具和输出形式
- 不把所有 Agent 逻辑堆成单一巨型执行器

### `providers/`

职责：

- 模型供应商适配
- 不同模型接口统一
- 请求参数与响应结构转换

约束：

- provider 只处理模型平台差异
- 不混入业务规则或主流程编排

### `tools/`

职责：

- Agent 可调用工具定义
- 工具输入输出结构
- 工具权限边界

约束：

- 工具应通过受控接口访问主业务后端或相关能力
- 不应把直接数据库写入暴露为默认工具能力

### `memory/`

职责：

- 会话记忆组织
- 摘要压缩
- 上下文裁剪
- 长短期上下文装配

约束：

- memory 只负责上下文组织
- 不应替代主业务后端的正式历史和正式状态

### `queue/`

职责：

- BullMQ 任务投递
- 队列消费
- 长任务处理
- 后台执行链路

约束：

- 队列是执行机制，不是业务真相
- 队列任务应可追踪和可恢复

### `contracts/`

职责：

- 与主业务后端的请求契约
- 事件契约
- 写回契约
- 输入输出数据结构约定

约束：

- contracts 必须清晰、稳定、可版本化
- 不通过隐式约定耦合 AI 层与业务层

## 5. Interaction With The Backend

AI 编排层与主业务后端之间应遵守以下边界：

主业务后端负责：

- 业务真相源
- 权限控制
- 核心领域状态
- 正式业务记录

AI 编排层负责：

- 读取所需上下文
- 组织模型调用和工具执行
- 生成建议、摘要、计划、候选同步项等中间结果
- 通过受控接口写回候选结果

强约束：

- AI 编排层不直接取代主业务后端
- 不直接复制主业务模块
- 不直接把未确认的 AI 输出写成最终业务状态

## 6. Tooling Rules

工具层应被视为 Agent 与系统交互的受控边界。

建议每个工具至少明确：

- 工具名称
- 输入结构
- 输出结构
- 适用场景
- 权限范围
- 是否允许写操作

禁止：

- 给 Agent 一个无限制的数据库直连工具
- 让工具绕过业务校验逻辑
- 用模糊的自然语言约定代替清晰的工具输入输出契约

## 7. Memory Rules

记忆能力应服务于编排，而不是取代业务系统。

推荐区分：

- 短期会话上下文
- 中期摘要上下文
- 长期可引用上下文

要求：

- 明确哪些记忆只是执行辅助
- 明确哪些内容需要回写主业务系统
- 不把“模型记得”当作系统级真实状态

## 8. Async Execution Rules

AI 层通常会承担较多异步处理任务，因此建议尽早按异步可演进方式组织。

适合异步的场景包括：

- 长摘要任务
- 多步 Agent 执行
- 外部信息同步
- 后台分析与候选生成

要求：

- 使用 BullMQ 一类机制承载异步任务
- 每个任务应可追踪状态
- 避免将长流程全部塞进单次 HTTP 请求

## 9. Candidate Write-Back Rules

AI 层生成的结果在写回系统前，推荐优先落为候选态，例如：

- draft
- suggestion
- candidate
- pending sync

这样做的目的：

- 保持业务真相可控
- 避免 AI 输出直接污染正式状态
- 便于后续人工确认、自动审核或业务流程吸收

## 10. Evolution Rules

AI 编排层演进时应遵守：

- 优先增强契约和边界，而不是直接增加隐式能力
- 优先增强可观察性，而不是只追求“能跑通”
- 优先把上下文来源、工具权限和写回路径显式化
- 在 Agent 类型、工具系统、记忆机制或写回机制发生明显变化时，先更新 OpenSpec

## 11. Implementation Checklist

后续新增 AI 能力时，至少检查以下问题：

- 该能力属于模型适配、编排、工具、记忆还是异步任务
- 是否错误承担了主业务后端职责
- 输出是否被误当作正式业务真相
- 工具权限是否过宽
- 写回路径是否经过受控契约
- 该变更是否需要先进入 OpenSpec
